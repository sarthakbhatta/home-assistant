# Room detail views — design

**Date:** 2026-07-19
**Status:** BUILT and live for the Office. Overview simplification (step 6) is
the only outstanding item.
**Scope of this spec:** the Office room only. Other rooms are explicitly deferred.

## Goal

Turn the Night Console **Plan** view from a status display into something you
operate. Today the whole house is one 1260×750 isometric image: the office
downlight fixtures render at roughly 4 px, which is far below a usable touch
target, so the plan can be looked at but not used.

The fix is two screens instead of one:

1. **Overview** — all six rooms, simplified, each tappable. Atmosphere and
   room-level status.
2. **Room screen** — one room rendered large, controls embedded in the 3D scene,
   with a drawer for the controls that a tap can't express.

## Decisions

| Decision | Choice | Why |
|---|---|---|
| Room screen style | Controls live **in the scene** — tap the lamp, tap the AC | The SmartThings look the user asked for |
| Advanced controls | A **drawer** that shrinks the scene to ~58% | Progressive disclosure: default to beauty, escalate to density on demand |
| Overview role | **Simplifies into a launcher** — keeps shell, furniture, light glow; drops per-device detail | Detail is illegible and untouchable at overview scale; it belongs on the room screen |
| First room | **Office only** | Richest room, and the one actually used. Proves the pattern end to end |

### Why not the other options

- **Overview unchanged** would keep 47 entities crammed into one image, half of
  them decoration nobody would tap because the room screen is better.
- **True dollhouse** (rooms in real relative position) is the best end state but
  needs the actual floor layout, which we don't have. It is *not* excluded —
  (b) can be re-arranged into it later without redrawing any room.
- **Bubble pop-up as the drawer** cannot shrink the scene. A standalone Bubble
  pop-up removes its own `ha-card` from the DOM and remounts `position: fixed`,
  so it occupies zero layout height and *overlays* the plan rather than
  displacing it. Overlay was offered and rejected.

## Mechanisms — what was proven, and how

All of the following was verified against the live instance
(HA OS 18.1 / Core 2026.7.2) on 2026-07-19, not inferred.

### Navigation — CONFIRMED

`tap_action: {action: navigate, navigation_path: ...}` on an ha-floorplan rule
performs soft SPA navigation.

Probe: rule 48 (`office-temp`) was temporarily repointed at
`/night-console/energy`. A flag was set on `window` before the click and
survived it, proving no page reload. Config was restored afterwards and verified
(`rule48tap == {action: 'more-info'}`, 49 rules intact).

Two traps:

- `action:` must be written explicitly. It defaults to `call-service`.
- A native SVG `<a href>` does **not** work. HA's click interception matches
  `tagName === "A"` and SVG anchors are lowercase; ha-floorplan also
  `preventDefault()`s every click on a ruled element.

### The drawer — CONFIRMED, and not the mechanism originally planned

**card-mod is not installed.** The Lovelace resource list is exactly:
mushroom, mini-graph-card, Bubble-Card, power-flow-card-plus, ha-floorplan.
Earlier notes in `DEPLOYED.md` claiming card-mod was present were wrong.
The card-mod `flex-basis` approach is therefore unavailable.

**Replacement, proven live:** the SVG lives inside `floorplan-element`'s
**shadow root**. Setting the SVG's own `height` reflows the card, which
displaces the sibling drawer card:

| Element | Drawer closed | Drawer open |
|---|---|---|
| `svg` | 843 px | 480 px |
| `floorplan-card` | 845 px | 482 px |
| `bubble-card` top | 909 px (below fold) | 546 px (visible) |

`height` transitions smoothly. Because the SVG's `<style>` block is generated by
`gen_iso.py`, we own that CSS outright — this needs **no new dependency**.

This is strictly better than the card-mod plan: card-mod could not have reached
into `floorplan-element`'s shadow root reliably anyway.

### In-scene control — CONFIRMED

Tapping an SVG element fires a service call. Verified by toggling
`input_boolean.office_drawer` from `off` to `on` via a tap on `office-temp`.
This is the same mechanism the existing 38 tap rules already use.

### Confirmed during the build

`floorplan.class_set` **does** work on the SVG root (`id="fp-root"`), and the
whole loop runs: handle tap → `input_boolean.office_drawer` → `class_set` on the
root → CSS height change → card reflow → controls slide up.

**`!important` is mandatory on the height rules.** ha-floorplan writes inline
`height: 100%; width: 100%; margin: auto; …` onto the SVG element and injects
its own rules into the same shadow root. Without `!important` the SVG sizes
itself from its aspect ratio and overflows the view badly — on a 1419 px-wide
panel the office room rendered ~1600 px tall and the page scrolled.

Measured live with the drawer open: computed height `383.672px`, which is
exactly 44vh of an 872 px viewport.

**Measurement caution:** `getBoundingClientRect()` sampled too soon after the
tap still reports the old height — the helper state has to round-trip through
HA before `class_set` fires. Check `getComputedStyle`, and allow ~2 s. This
briefly looked like a failure during the build and was not one.

## Constraints discovered

These are load-bearing. Violating any of them produces a silent failure.

1. **The room groups currently have no ids.** The SVG exposes only `bg`,
   `poolg`, `amb`, `office-temp`, `mon-screen`, `mon-body`, `tv-body`,
   `motion-fx`, `siren.hub`, and the 47 entity ids. Tap-to-enter requires ids on
   the room `<g>` elements.
2. **Saving the dashboard config forces a full SVG re-fetch.** Observed live —
   the floorplan disappeared for ~4 s after a save. This is why the drawer must
   **not** be a conditional-card swap: every toggle would blank the scene.
3. **A panel view renders only `cards[0]`.** Sibling cards are never
   instantiated and fail silently. The `vertical-stack` wrapper is mandatory,
   not stylistic.
4. **Every view needs an explicit `type:`.** Pre-existing project rule; a
   missing type causes fully-blank *or* partially-blank renders.
5. **`image:` must be a plain string path.** The `{location, cache}` object form
   silently does nothing on ha-floorplan v1.1.5.
6. **Sections views cannot express "100% of view height."** Stay on
   `type: panel` for anything full-bleed.
7. **No full-size transparent background rect**, ever — ha-floorplan forces
   `pointer-events: all !important` on `svg *`, so such a rect swallows every
   tap. Checked: `bg` is a `linearGradient` def, so this is currently fine.

## Architecture

### `gen_iso.py` changes — one pass, three purposes

- Emit an `id` on each room `<g>` (`room-office`, `room-bedroom`, …) so rooms
  can be tapped.
- Emit an `id` on the SVG root so the drawer class has a target.
- Add drawer CSS to the generated `<style>`: a `drawer-open` class on the root
  that sets the reduced height, plus the transition.
- Add a **room-only render mode** that emits a single room at a larger scale on
  its own canvas. The `Room` class already uses local coordinates, so this is a
  scale and canvas change, not a redraw.
- **Simplify the overview**: keep shell, furniture silhouette and light glow;
  drop per-device detail. "Room-level status" means exactly three things per
  room, and nothing else:
  1. **Is any light on** — the room's floor glow, already implemented.
  2. **One number**, where the room has a meaningful one (Office: temperature;
     others: omitted rather than invented).
  3. **An alert dot**, only when something in that room needs attention
     (terrace door ajar, borewell overheat/overload).

  Everything else — individual fixtures, plugs, the monitor, the TV, the desk
  chair — is silhouette only, with no entity binding at overview scale.

Two SVGs are produced: `overview.svg` and `room-office.svg`.

**Naming, because the existing names are actively misleading.** The file
currently deployed at `/config/www/floorplan/office_iso.svg` is *not* the
office — it is the whole-house six-room overview. And `office.svg` on the HA box
is the superseded flat proof-of-concept. So neither existing name is reusable:
`office.svg` would collide with a real file, and `office_iso.svg` means the
opposite of what it says. New files get unambiguous names, and the two legacy
files are left in place until the new ones are verified, then removed.

### Dashboard

**Overview** — existing `plan` view, `type: panel`, one `floorplan-card`. Rules
gain one `navigate` per room. Only Office resolves to a room screen in this
pass; the other five keep their current behaviour.

**Office room screen** — new view, `type: panel`, one `vertical-stack`:

- **card 0** — `custom:floorplan-card` rendering `office.svg`. Rules for
  in-scene control (tap to toggle, hold for more-info), plus one rule that
  `class_set`s `drawer-open` on the root from `input_boolean.office_drawer`.
- **card 1** — ordinary Bubble cards: brightness sliders, AC setpoint, live
  watts. Explicitly **not** `card_type: pop-up`.

### State

`input_boolean.office_drawer` (already created). Toggled by a handle element in
the scene.

**Known cost:** this is global state. Opening the drawer on the wall tablet
opens it on the phone too, and every toggle writes to the recorder. Accepted for
now because it is the only mechanism that can drive a layout change. If it
becomes annoying, the fallback is one helper per device.

## Build order

Stop at the first failure; each step is independently verifiable.

1. `gen_iso.py`: room ids + SVG root id + drawer CSS. Regenerate the overview,
   deploy, confirm the plan still renders and rooms are tappable.
2. Add `navigate` on the office room → new empty room view. Confirm the
   transition.
3. `gen_iso.py`: room-only render mode. Generate `room-office.svg`, deploy,
   confirm it renders standalone via the preview harness.
4. Build the room view: floorplan + a **cut-down** rule set (2–3 rules, not the
   full set) + one Bubble card. Confirm the drawer opens and animates.
5. Full rule set and the animation pass.
6. Simplify the overview art.

Deploy SVGs via the LAN HTTP method (`python3 -m http.server` on the Mac,
`curl` from the HA box, verify md5). Typing base64 into the ingress terminal
corrupts payloads and must not be used. Cache-bust by bumping the `?v=` query
string, since the string form of `image:` has no cache option.

## Out of scope

- Bedroom, Staircase — same pattern, deferred until Office is proven.
- Living, Terrace, Garden — these need *different* patterns (media surface,
  camera feed, watering schedule). Forcing them into this template would be
  worse than designing them separately.
- The dollhouse re-arrangement.
- The "Flight Deck" wall-tablet dashboard.

## Risks

| Risk | Mitigation |
|---|---|
| `class_set` on the SVG root doesn't work | Falls back to a wrapper `<g>` with the id, which is known to work |
| Blur/filter re-rasterisation makes the height animation janky | Fall back to `transform: scale()`, which is GPU-composited |
| Hairline strokes vanish at 58% | `non-scaling-stroke` is available; check visually at the reduced size |
| Room screen looks sparse | Office is the richest room, so this is the best case — if it looks thin here, the pattern is wrong and we should learn that early |
